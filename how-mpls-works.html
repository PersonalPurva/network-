🧭 The one-sentence mental model

MPLS = "route once at the edge, switch on a label through the core." Instead of every router re-reading the IP header and doing a slow, longest-prefix routing lookup, the first router picks the whole path and slaps a short label on the packet. Every core router then just reads that label, swaps it, and forwards — a fast, exact-match table lookup. The label is a pre-paid ticket that says "put me on train 17"; the core never re-checks your final address.

1

Why MPLS exists — the problem with plain IP

On the normal internet, every router independently reads the destination IP and does a longest-prefix match against a huge routing table to decide the next hop. That has three pain points on a carrier backbone:

Repeated lookups

Every hop re-parses the IP header and searches the full table. Wasteful when the path is already known.

No path control

Plain IP always takes the "shortest" route. You can't say "send this traffic the long way to balance load" — no traffic engineering.

No clean VPNs

Keeping hundreds of customers' overlapping private IPs separate over one shared backbone is hard with IP alone.

MPLS's answer

Do the expensive IP decision once, at the edge, and encode the result as a label. The core switches on labels (simple exact match), can be steered onto any path the operator chooses, and can stack a second label to keep each customer's traffic in its own private tunnel. That's the whole value proposition: speed, control, and isolation.

Note: modern core silicon does IP lookups fast too, so raw speed is less of an argument today — traffic engineering and VPNs are why MPLS still dominates telecom/ISP backbones, and why ISRO's backbone in PS#13 is modelled as MPLS.

2

The label & the "layer 2.5" shim

MPLS inserts a 32-bit shim header between the layer-2 frame header (Ethernet) and the layer-3 packet (IP) — which is why MPLS is nicknamed "layer 2.5."

The 32-bit MPLS header, field by field

Label

20 bits

TC

3

S

1

TTL

8

  • • Label (20 bits) — the value core routers match on. ~1 million possible; values 0–15 reserved.
  • • TC / EXP (3 bits) — Traffic Class (old name "EXP"). Carries QoS priority — voice vs bulk.
  • • S — Bottom of Stack (1 bit) — 1 means "this is the last/innermost label." Lets routers stack multiple labels (VPNs need two).
  • • TTL (8 bits) — same idea as IP TTL: decremented each hop, kills looping packets.

Reserved labels to know

0 = explicit null (IPv4), 2 = explicit null (IPv6), 3 = implicit null (triggers PHP — section 4).

Label stack

Labels can be pushed on top of each other. The S bit marks the bottom. Outer label = transport across the core; inner label = which VPN/service. This two-label stack is the heart of MPLS VPNs.

3

The devices — edge vs core

An MPLS network has two router roles. Getting these names straight makes everything else click.

EDGE

LER / PE router

Label Edge Router · Provider Edge

Sits at the border of the MPLS cloud. Does the heavy IP work: classifies the packet, pushes the first label (ingress) or pops it and returns to plain IP (egress). The smart router.

CORE

LSR / P router

Label Switching Router · Provider

Lives inside the cloud. Only ever swaps labels using a fast table — never looks at the customer IP at all. The dumb-but-fast router.

CUSTOMER

CE router

Customer Edge

The customer's own router that hands plain IP to the PE. It has no idea MPLS exists — that's the point of a provider VPN.

Mnemonic: Provider = Pure core (only labels). PE = Provider Edge (the translator). CE = Customer's box. Traffic path is always CE → PE → P → P → PE → CE.

4

The three operations: push, swap, pop

Everything MPLS does to a packet is one of three label operations. First, one key idea:

FEC — Forwarding Equivalence Class

A FEC is a group of packets that should all be treated the same way and take the same path (e.g. "everything going to network 10.0.5.0/24"). At the ingress, the PE decides which FEC a packet belongs to once, and maps that FEC to a label. All packets in a FEC ride the same LSP (label-switched path).

A packet crossing the backbone

Ingress PE

PUSH. Reads the IP, picks the FEC, pushes label 17. Plain IP packet → labelled packet.

Core P #1

SWAP. Sees label 17 in its table → "swap to 42, send out port 3." Never reads the IP.

Core P #2

SWAP. Label 42 → 88, out port 1.

Penult. P

POP (PHP). With penultimate hop popping, the second-to-last router removes the label so the egress PE gets a clean packet and doesn't do a double lookup. Signalled by reserved label 3.

Egress PE

Forward as IP. Sends the original packet to the customer (CE). Back to normal routing.

PUSH

Add a label (ingress, or entering a VPN tunnel).

SWAP

Replace the top label with a new one (every core hop).

POP

Remove the top label (egress / penultimate hop).

5

How labels get agreed — LDP & the tables

Labels are only locally meaningful, so neighbours must agree "when I say label 17, I mean this FEC." A label distribution protocol spreads that agreement across the network.

LDP — Label Distribution Protocol

The common one. Each router advertises its label-to-FEC bindings to neighbours, so a full path of labels forms automatically, following the normal IP routing (it just labels the existing shortest path). Simple, no path control.

RSVP-TE — for traffic engineering

Used when you want a specific path with reserved bandwidth (not just shortest). It signals an explicit route hop by hop and reserves resources. This is what enables MPLS-TE and Fast ReRoute (section 8). Modern networks increasingly use Segment Routing instead (below).

The three tables inside a router

RIB / FIB

The normal IP routing / forwarding table — where IP destinations go.

LIB

Label Information Base — all label bindings learned from all neighbours (the full catalogue).

LFIB

Label Forwarding Information Base — the chosen in-label → out-label + port entries actually used to switch packets. The hot path.

Flow: LDP fills the LIB → the best bindings are installed into the LFIB → packets are switched by LFIB lookups. In your NOC, watching these tables change is a security signal — a label binding that shouldn't exist is an attack.

6

LSP — the label-switched path

What an LSP is

An LSP is the end-to-end tunnel a labelled packet follows from ingress PE to egress PE — the chain of swap operations. It is unidirectional: traffic the other way uses a separate LSP.

Think of it as a one-way virtual circuit built out of labels, laid over the shared physical network.

Signalled vs static

Signalled LSPs are built automatically by LDP/RSVP-TE (normal). Static LSPs are hand-configured (rare, for special control). An LSP that appears without a legitimate signalling trigger is exactly the kind of anomaly a NOC should flag — see the honey-LSP threat.

7

MPLS VPNs — the killer feature

The reason carriers love MPLS: one physical backbone can carry hundreds of customers, each feeling like they have a private network — even if two customers use the exact same private IP range (10.0.0.0/8). Here's the machinery.

L3VPN — the common one

The provider routes IP for the customer. Built from four pieces:

  • • VRF (Virtual Routing & Forwarding) — each customer gets its own private routing table on the PE, so their routes never mix.
  • • RD (Route Distinguisher) — an 8-byte value prepended to each route to make overlapping IPs globally unique (customer-A's 10.1.1.0 ≠ customer-B's).
  • • RT (Route Target) — a BGP tag controlling which VRFs import/export which routes — this is what defines VPN membership (and, if misconfigured, leaks between customers).
  • • MP-BGP — carries these VPN routes (VPNv4) between PEs across the core.

The two-label stack in action

VPN traffic carries two labels:

Outer label — transport: gets the packet PE→PE across the core (LDP)
Inner label — VPN: tells the egress PE which customer VRF to deliver to (MP-BGP)

Core P routers only read the outer label — they have no clue which customer it is. The egress PE pops the outer, reads the inner, and drops the packet into the right VRF. Isolation achieved.

L2VPN — extending a LAN

Instead of routing, the provider carries the customer's raw layer-2 frames across the MPLS core, so two distant sites feel like one LAN. Point-to-point is a pseudowire; multipoint is VPLS. Handy context, but L3VPN is the one your PS#13 model centres on (route leakage threat).

8

Traffic engineering, QoS & fast reroute

MPLS-TE

Steer traffic onto specific paths (not just shortest) to balance load and honour bandwidth guarantees. Built with RSVP-TE, using link-state info (OSPF/IS-IS with TE extensions) to compute a path that fits.

QoS via the TC bits

The 3 TC/EXP bits mark priority so voice/critical traffic gets low latency while bulk data waits. Carriers sell SLAs on this. Relevant to detecting traffic-flooding attacks that starve priority queues.

FRR — Fast ReRoute

Pre-computed backup LSPs let traffic switch to an alternate path in < 50 ms when a link fails — faster than any routing reconvergence. Critical for a resilient backbone.

Modern evolution — Segment Routing (SR-MPLS): encodes the whole path as a list of labels ("segments") pushed by the ingress, so core routers need no per-flow signalling state (no LDP/RSVP). Simpler, more scalable; increasingly the default on new backbones. Worth a one-line mention in your NOC defence — it changes how labels are distributed but not the label-switching fundamentals above.

🛰️ Watching MPLS — the air-gapped NOC (your BAH model)

Now the fundamentals connect to PS#13. A NOC (Network Operations Centre) is the room/software that watches the whole backbone. Your model makes it an offline AI copilot: it predicts faults before they happen and flags security threats — all without internet, because a national telecom/space backbone must stay air-gapped.

Three telemetry planes (what the NOC ingests)

1 · Control plane

gNMI + OpenConfig models stream device state (labels, LSPs, interfaces, CPU). Vendor-neutral YANG telemetry — the modern replacement for SNMP polling.

2 · Routing plane

BMP (BGP Monitoring Protocol) streams every BGP/VPN route change — the feed that catches hijacks and VPN route leaks.

3 · Optical plane

DWDM optical metrics (pre-FEC BER, power levels) — the physical-layer feed that predicts fibre degradation before a link drops.

The four-stage cascade detector

  1. Statistical — cheap thresholds / z-scores catch obvious spikes.
  2. Isolation Forest — unsupervised outlier detection on multi-metric data.
  3. LSTM autoencoder — learns "normal" time patterns; high reconstruction error = anomaly.
  4. Change-point — detects a regime shift (the moment behaviour changed).

Cascade = cheap filters first, expensive models only on survivors. Efficient on a no-GPU / offline box.

Why offline + trustworthy

  • • Local quantized LLM + RAG — explains alerts in plain language and cites runbooks, with no cloud call (data never leaves the air gap).
  • • Tamper-evident forensic log — SHA-256 hash-chained incident ledger; any edit breaks the chain (verifiable).
  • • MITRE ATLAS hardening — protects the AI itself: artifact-integrity hashing, data-poisoning drift checks vs a golden baseline.
  • • Predict-before-fail (ETTF) — extrapolates optical BER to the FEC threshold with a calibrated prediction interval (estimated time-to-failure).

🛡️ Keep it measured for judges: this is a proof-of-concept that detects modelled threats from telemetry, with stated limits (synthetic/emulated data, signature + anomaly based). Don't claim it beats commercial NMS — claim it's an offline, explainable, tamper-evident copilot for a specific threat set. See your no-overclaiming rule.

🛡️ The six MPLS threats, mapped to the fundamentals

Each PS#13 threat attacks one MPLS mechanism you just learned. This is the table to be able to defend cold in the finale.

ThreatAttacks which mechanismTelemetry signal that catches it
Label spoofingForged MPLS labels injected to misroute / bypass VPN isolation (§2, §4)Unexpected LFIB entries / labels with no signalling origin (gNMI)
BGP hijackFalse route advertisements steal/blackhole traffic (§7 MP-BGP)Sudden origin/AS-path change in the BMP feed
BFD session attackForging/dropping the fast liveness protocol to trigger false failovers (§8 FRR)Abnormal BFD flap rate / timing anomalies
VPN route leakageMis-set Route Targets leak one customer's routes into another VRF (§7 RT/VRF)Route appearing in a VRF whose RT policy forbids it (BMP)
Traffic floodingVolumetric overload starving priority queues / links (§8 QoS)Throughput + queue-drop spikes across the cascade detector
Honey-LSP tripwireDefensive: a decoy LSP no legit traffic should touch (§6 LSP)Any packet/label activity on the honey-LSP = confirmed intrusion

Notice the pattern: every threat is a lie told in the control/routing plane — a fake label, a fake route, a fake liveness signal — and every defence is watching the telemetry for a state that shouldn't exist. That's the whole thesis of your NOC: you can't stop a compromised device from lying, but you can detect when the network's observed state diverges from what legitimate signalling would produce. Observability is the security control.

Quick cheat sheets

MPLS acronyms

  • LER / PE — edge router (push/pop)
  • LSR / P — core router (swap)
  • CE — customer edge
  • FEC — forwarding equivalence class
  • LSP — label-switched path
  • LDP / RSVP-TE — label distribution
  • LIB / LFIB — label tables
  • VRF / RD / RT — VPN separation
  • MP-BGP — carries VPN routes
  • PHP — penultimate hop popping
  • FRR — fast reroute (<50 ms)

The packet's journey (recite)

  1. CE sends plain IP to ingress PE
  2. PE picks FEC, pushes label(s)
  3. Core P routers swap the outer label
  4. Penultimate P pops (PHP)
  5. Egress PE reads inner VPN label → VRF
  6. Delivers plain IP to destination CE

NOC / BAH terms

  • gNMI / OpenConfig — control telemetry
  • BMP — BGP route monitoring
  • DWDM / pre-FEC BER — optical health
  • BFD — fast failure detection
  • ETTF — estimated time-to-failure
  • MITRE ATLAS — AI-system threat model
  • Air-gap — no internet, offline AI
